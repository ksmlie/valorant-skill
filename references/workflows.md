# 工作流

## 1. 选手查询

用户询问某位选手。

流程：
- 识别选手名称或别名
- 映射到标准选手 id
- 获取选手资料
- 如有可用数据，再获取近期结果
- 返回摘要和结构化字段

## 2. 战队查询

用户询问某支战队。

流程：
- 识别战队名称或别名
- 映射到标准战队 id
- 获取战队资料
- 如有可用数据，再获取近期比赛结果
- 返回当前状态摘要

## 3. 比赛结果查询

用户询问某场比赛谁赢了。

流程：
- 识别战队、赛事或日期（如有）
- 查找对应的比赛记录
- 返回胜者、比分，以及可用时的地图摘要

## 4. 赛程查询

用户询问今天、即将进行的比赛，或某个日期范围。

流程：
- 规范化日期表达
- 筛选赛程条目
- 按时间顺序返回比赛

## 5. 赛事汇总查询

用户询问某个赛事的完整参赛队伍、阶段结构或最终排名。

### 必须先做的事

- 先在赛事总页中识别所有子页面链接。
- 再逐个访问所有子页面。
- 最后把所有子页面的结果合并后再统计。

### 已结束赛事

- `Swiss Stage`、`Group Stage`、`Playoffs` 等阶段页中出现的队伍，都应纳入赛事完整结果。
- 阶段页里的名次标号应视为该赛事的最终排名信息。
- 汇总时必须按**所有子页面合并后的结果**统计，不可只看单页。

### 未开始或进行中的赛事

- 不关注最终排名。
- 仍然必须先收集所有子页面链接，再汇总所有子页面中的参赛队伍。
- 去重后得到完整参赛队伍集合。
- 不要只统计当前页面可见内容。

### 额外约定

- 如果某些队伍在某些子页面里暂时没有排名，可以留空。
- 但队伍本身如果在任一子页面中出现，就应计入赛事汇总。

## 一致性规则

- 优先使用标准 id，而不是展示名称。
- 保留原始用户输入，便于追踪。
- 返回结构化的空结果，不要自由猜测。
