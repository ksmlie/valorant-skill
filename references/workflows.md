# 工作流

## 1. 选手查询

用户询问某位选手。

流程：
- 识别选手名称或别名
- 映射到标准选手 id
- 获取选手资料
- 如有可用数据，再获取近期结果
- 返回摘要和结构化字段

## 2. 战队查询

用户询问某支战队。

流程：
- 识别战队名称或别名
- 映射到标准战队 id
- 获取战队资料
- 如有可用数据，再获取近期比赛结果
- 返回当前状态摘要

## 3. 比赛结果查询

用户询问某场比赛谁赢了。

流程：
- 识别战队、赛事或日期（如有）
- 查找对应的比赛记录
- 返回胜者、比分，以及可用时的地图摘要

## 4. 赛程查询

用户询问今天、明天、某个日期，或某个日期范围的比赛。

### 查询流程

1. 先判断用户是在查：
   - 单日赛程（如 `today`、`tomorrow`、`6-14`、`2026-06-14`）
   - 赛事内赛程（如 `masters-london-2026`、`playoff`、`swiss stage`）
   - 全站赛程（如“今天有哪些比赛”）
2. 规范化日期表达，只根据**目标日期本身**解析，不要依赖 `Yesterday`、`Today` 这些页面装饰文案来推断日期。
3. 如果是赛事内赛程：
   - 先读取对应赛事总页或赛事子页
   - 按日期、阶段、状态筛选比赛
4. 如果是全站赛程：
   - 读取 VLR 全站赛程页
   - 按页面展示日期分组
   - 只返回目标日期下的比赛
5. 按时间顺序返回比赛，并保留：
   - `match_id`
   - `date`
   - `time`
   - `stage`
   - `status`
   - `team_a` / `team_b`

### 解析规则

- 日期标题里的 `Today`、`Yesterday`、`Tomorrow` 仅作辅助标签，不参与日期归属判断。
- 当页面同一日期下出现多场同一时间的比赛时，应全部返回。
- 如果用户输入的是短日期（如 `6-14`），默认补全年份并按本地默认年份解析。
- 若目标日期在当前数据源中无比赛，返回结构化空结果，不要猜测。

## 5. 赛事汇总查询

用户询问某个赛事的完整参赛队伍、阶段结构或最终排名。

### 查询规则

1. 先读取赛事总页，确认赛事基本信息。
2. 查找总页中所有可用的子页面链接，例如 `swiss-stage`、`playoffs`、`group-stage`。
3. 逐个打开每个子页面，读取其中展示的队伍、排名、阶段结果和其他可见信息。
4. 对于已经结束的赛事：
   - 合并所有子页面后再统计参赛队伍。
   - 子页面中展示的排名标号，视为该赛事的最终排名信息。
5. 对于未开始或进行中的赛事：
   - 不关注最终排名。
   - 只统计所有子页面合并后的参赛队伍集合。
   - 如果某些队伍暂未有最终名次，可留空。
6. 永远不要只根据单个子页面就做最终汇总。

### 必须先做的事

- 先在赛事总页中识别所有子页面链接。
- 再逐个访问所有子页面。
- 最后把所有子页面的结果合并后再统计。

### 已结束赛事

- `Swiss Stage`、`Group Stage`、`Playoffs` 等阶段页中出现的队伍，都应纳入赛事完整结果。
- 阶段页里的名次标号应视为该赛事的最终排名信息。
- 汇总时必须按**所有子页面合并后的结果**统计，不可只看单页。

### 未开始或进行中的赛事

- 不关注最终排名。
- 仍然必须先收集所有子页面链接，再汇总所有子页面中的参赛队伍。
- 去重后得到完整参赛队伍集合。
- 不要只统计当前页面可见内容。

### 额外约定

- 如果某些队伍在某些子页面里暂时没有排名，可以留空。
- 但队伍本身如果在任一子页面中出现，就应计入赛事汇总。

## 一致性规则

- 优先使用标准 id，而不是展示名称。
- 保留原始用户输入，便于追踪。
- 返回结构化的空结果，不要自由猜测。
