# Valorant 职业赛事实况查询技能

## 目的

本技能用于解析用户对《无畏契约》职业赛事的自然语言提问，并将其稳定映射到对应的数据查询脚本。核心目标不是生成主观解读，而是输出可追踪、可验证、便于下游系统消费的结构化结果。

当前技能重点支持以下四类查询：

1. 选手信息与近期战绩
2. 战队信息与近期战绩
3. 比赛结果与交手记录
4. 赛程与未来赛事

## 设计原则

- 以事实查询为核心，不做推断性扩展。
- 优先返回结构化结果，保证字段稳定、语义明确。
- 遇到多种可能路由时，优先选择最具体、信息损失最小的脚本。
- 找不到精确匹配时，必须返回结构化 `not_found`，禁止猜测。
- 保持实现轻量、模块化，便于后续逐步扩展。

## 数据源策略

- 主数据源仅使用 `VLR.gg`。
- 不引入其他外部站点作为事实来源。
- 所有结果都应以真实页面数据为准，避免使用臆测或拼接结论。
- 当页面结构变化、字段缺失或抓取失败时，允许使用本地静态数据兜底，但必须明确标记来源。

### 选手数据约束

选手信息必须遵循 `references/vlr-data-refresh.md` 中定义的刷新顺序：

1. 先查询战队页中的 `Current Roster`
2. 再访问选手主页
3. 最后补充 `?timespan=90d` 维度的数据

这是为了保证阵容、身份和近期表现之间的关联关系一致，避免单页信息导致的误判。

### 赛事数据约束

赛事查询必须先读取赛事总页，再汇总其子页面信息，最后归纳出队伍、阶段、排名和结果。禁止直接依据单个子页面推出完整赛事结论。

## 路由总览

下面是技能层面的核心路由约定：

- 战队列表与战队概览 -> `scripts/valorant-teams.js`
- 赛程与今日 / 未来比赛 -> `scripts/valorant-schedule.js`
- 比赛结果与交手记录 -> `scripts/valorant-match.js`
- 战队详情与近期状态 -> `scripts/valorant-team.js`
- 选手详情与近期结果 -> `scripts/valorant-player.js`

## 查询优先级

1. 先识别问题意图，再映射脚本。
2. 如果用户同时在问“对象 + 结果 + 赛程”等复合问题，优先处理最确定的主查询，再按需补充次级信息。
3. 如果请求范围过大，返回简洁摘要，并用结构化字段承载核心事实。
4. 对于存在歧义的简称、别名或缩写，优先使用稳定的归一化规则。

## 输出规范

所有脚本都应尽量返回稳定的 JSON 风格结构，以便被上层程序或其他 AI 组件直接消费。

推荐顶层字段如下：

- `query_type`
- `matched_script`
- `normalized_query`
- `filters`
- `result`
- `source`
- `notes`

### 字段约定

- `query_type`：本次查询的意图分类
- `matched_script`：实际命中的脚本名称
- `normalized_query`：归一化后的查询表达
- `filters`：用于查询的关键过滤条件
- `result`：最终业务结果
- `source`：数据来源与解析模式
- `notes`：补充说明、异常提示或边界条件

### 返回要求

- 使用稳定字段名，不要随意扩展顶层结构。
- 找不到结果时返回 `status = "not_found"` 或等价结构。
- 缺失字段应显式返回 `null`，不要用空字符串冒充有效值。
- 数据来源不确定时必须保留来源说明，不能默认为已验证事实。

## 数据建模建议

- `data/teams.json`：存储战队静态身份、别名与基础映射。
- `data/events.json`：存储赛事身份与战队参赛关系。
- 后续可按需扩展 `players.json`、`matches.json`、`schedule.json` 等数据文件。
- 静态数据应作为索引和兜底层，而不是替代真实页面数据。

## 实现要求

- 先做确定性匹配，再考虑模糊匹配。
- 别名归一化要尽量前置，避免在各脚本中重复实现。
- 路由逻辑应简单明确，减少分支耦合。
- 对外输出要强调“事实可追踪”，而不是“答案看起来像对的”。
- 当无法稳定判定时，宁可返回 `not_found`，也不要输出可能错误的内容。

## 建议的结果结构

### 战队查询

适用于战队资料、阵容、状态、近期战绩等场景。

- `result.team`
- `result.roster`
- `result.recent_results`
- `result.recent_form`
- `result.stats`

### 选手查询

适用于选手资料、所在战队、近期表现、统计摘要等场景。

- `result.player`
- `result.team`
- `result.recent_matches`
- `result.stats`

### 比赛查询

适用于比赛胜负、比分、对阵、历史交手等场景。

- `result.match`
- `result.teams`
- `result.score`
- `result.winner`
- `result.map_results`

### 赛程查询

适用于今日赛程、未来赛程、某战队赛程等场景。

- `result.schedule`
- `result.events`
- `result.matches`
- `result.timezone`

## 典型问题示例

- 今天有哪些比赛
- 某战队最近状态怎么样
- 某选手的资料和战绩
- 某场比赛谁赢了
- 某战队的赛程
- 某战队有哪些选手
- 某选手最近几场表现如何

## 质量标准

这个技能文件应始终体现以下标准：

- 术语准确，避免口语化和模糊表达。
- 结构清晰，适合工程协作与后续维护。
- 约束明确，减少实现偏差。
- 说明充分，但不过度冗长。
- 以“查询系统规范”而非“产品介绍文案”的方式编写。

## 维护建议

当脚本能力或数据结构发生变化时，应同步更新本文件，确保技能描述、路由规则和输出约定始终一致。

如果未来增加新的查询能力，建议优先补充：

1. 新的路由入口
2. 新的输出字段说明
3. 新的数据源约束
4. 新的错误与兜底规则

这样可以保持技能文档与实现之间的长期一致性。
